# Задание на разработку системы ИИ-прогнозирования академической задолженности

**Название проекта:** Разработка прототипа ИИ-ассистента для прогнозирования рисков академической задолженности студентов.

**Цель проекта:** Разработать и верифицировать программный модуль (API или веб-интерфейс), который на основе исторических данных об успеваемости студентов будет автоматически прогнозировать вероятность возникновения академических задолженностей (не сданных зачётов/экзаменов) на предстоящей сессии, тем самым позволяя деканату и кураторам своевременно выявлять студентов группы риска и принимать превентивные меры.

**Проблема (обоснование):** В образовательном процессе деканаты и кураторы сталкиваются с необходимостью ручного мониторинга успеваемости сотен студентов. Прогнозирование потенциальных задолженностей на следующей сессии осуществляется эмпирически, без использования данных, что приводит к несвоевременному выявлению проблемных студентов и снижению общей успеваемости. Существует накопленный массив данных о текущей успеваемости, посещаемости, оценках за прошлые семестры и характеристиках дисциплин, что создаёт возможность для применения методов машинного обучения с целью автоматизации прогнозирования и повышения эффективности работы деканата.

---

## Общие требования к решению:

1. **Источник данных:**
   - Решение должно уметь принимать на вход Excel-файлы с данными об успеваемости студентов (аналогичные `example.xlsx`).
   - Дополнительно могут использоваться данные о посещаемости, нагрузке преподавателей, сложности дисциплин (если доступны).

2. **Целевая переменная (таргет):**
   - Количество задолженностей (не сданных зачётов/экзаменов) на **следующей сессии** относительно текущего семестра.
   - **Категории прогноза (многоклассовая классификация):**
     - `0` — нет задолженностей
     - `1` — 1 задолженность
     - `2` — 2 задолженности
     - `3+` — 3 и более задолженностей

3. **Ядро системы:** Разработать модель машинного обучения для **многоклассовой классификации**, используя для обучения исторические данные об успеваемости, извлечённые из `data.xlsx` и аналогичных файлов. Модель должна предсказывать категорию риска для каждого студента перед началом сессии.

4. **Результат:** На выходе система должна выдавать:
   - Список студентов, отсортированный по убыванию прогнозируемого количества задолженностей.
   - Для каждого студента — прогнозируемую категорию и вероятности по каждому классу.
   - Возможность экспорта результатов в Excel для дальнейшей работы деканата.

5. **Интерфейс:** Разработать простой веб-интерфейс (или командную строку) для:
   - Загрузки нового файла с данными об успеваемости.
   - Отображения ранжированного списка студентов с прогнозами.
   - Визуализации распределения прогнозов (графики, диаграммы).

6. **Критерии приемки:**
   1. **Работоспособность:** Приложение успешно обрабатывает новый файл и выводит прогнозы для всех студентов.
   2. **Качество модели:** Метрика `macro F1-score` на валидационной выборке не ниже **0.70**. Это гарантирует сбалансированное качество предсказаний по всем категориям риска.
   3. **Интерпретируемость:** Модель должна предоставлять важность признаков (feature importance) для объяснения прогнозов.
   4. **Документирование:** Наличие чёткой документации по установке, настройке и использованию приложения.
   5. **Воспроизводимость:** Код должен быть хорошо структурирован и содержать все необходимые файлы для повторного обучения модели.

## Этапы и задачи проекта:

### 1а. Этап: Анализ данных и проектирование признаков

**Задача:** Провести исследовательский анализ данных (EDA) для предоставленных файлов (на примере [данные/data.xlsx](данные/data.xlsx)).

- Изучить структуру и типы данных.
- Проанализировать динамику успеваемости студентов по семестрам.
- Выявить закономерности:
  - Какие дисциплины чаще всего вызывают задолженности?
  - Как связаны оценки в предыдущем семестре с количеством задолженностей в следующем?
  - Влияет ли тип дисциплины (экзамен/зачёт) на вероятность задолженности?
  - Есть ли сезонность (I vs II полугодие)?
- Проанализировать данные на наличие пропусков и выбросов (например, "Неявка", "Неудовлетворительно").

**Задача:** Спроектировать и создать вектор признаков (Feature Engineering) для обучения модели.

- **Агрегированные признаки по студенту за семестр:**
  - Количество оценок "Отлично", "Хорошо", "Удовлетворительно", "Неудовлетворительно", "Неявка".
  - Средний балл за семестр (перевод оценок в числовой эквивалент).
  - Количество зачётов/экзаменов в семестре.
  - Доля "зачтено" среди зачётов.
  - Наличие повторяющихся дисциплин (пересдачи).

- **Временные признаки:**
  - Номер семестра (1, 2, 3, ...).
  - Общее количество семестров обучения.
  - Тенденция изменения среднего балла (сравнение с предыдущим семестром).

- **Признаки по дисциплинам:**
  - Средняя успеваемость группы по дисциплине (сложность дисциплины).
  - Тип дисциплины (гуманитарная, техническая, естественнонаучная и т.п.).

- **Целевая переменная (`target`):**
  - Количество задолженностей в **следующем семестре** (сдвиг по времени).
  - Учитывать, что для последнего семестра целевая переменная отсутствует (эти данные исключаются из обучения или используются как тестовая выборка).

**Результат:** Очищенный и размеченный датасет для обучения, где каждая строка — это студент за определённый семестр, с агрегированными признаками и целевой переменной (количество задолженностей в следующем семестре).

### 1б. Этап: Разработка и обучение моделей

**Задача:** Разработать и обучить несколько моделей машинного обучения.

- Разделить датасет на обучающую и тестовую выборки (например, 80/20) с учётом временной последовательности (более ранние семестры — в обучение, более поздние — в тест).
- Обучить как минимум 3 модели для сравнения:
  - **Logistic Regression** (как бейзлайн).
  - **Random Forest** (для интерпретируемости).
  - **XGBoost / CatBoost** (для максимальной точности).
- Провести кросс-валидацию для настройки гиперпараметров (с учётом временной структуры — TimeSeriesSplit).
- Применить методы балансировки классов (class_weight, SMOTE), если классы несбалансированы.

**Задача:** Оценить качество моделей.

- Использовать метрики:
  - `accuracy` — общая доля правильных ответов.
  - `precision`, `recall`, `F1-score` — для каждого класса отдельно.
  - `macro` и `weighted` F1-score — для общей оценки.
- Особое внимание уделить **recall для класса "3+"** (высокий риск) — минимизация пропуска студентов с наибольшим риском отчисления.

**Результат:** Выбранная лучшая модель, сохранённая в файл (например, `model.pkl`), с описанием её характеристик и обоснованием выбора.

### 2. Этап: Разработка бекенд-приложения и веб-интерфейса

**Задача:** Создать веб-приложение для взаимодействия с моделью.

- Разработать минималистичный фронтенд на **Flask/FastAPI** с формой для загрузки Excel-файла.
- Реализовать бекенд, который:
  1. Принимает и обрабатывает загруженный файл.
  2. Применяет к данным ту же логику предобработки и агрегации признаков, что использовалась при обучении.
  3. Загружает сохранённую модель и делает предсказание для каждого студента.
  4. Выводит на экран:
     - Таблицу со студентами, отсортированными по убыванию прогнозируемого количества задолженностей.
     - Для каждого студента — вероятности по каждому классу.
     - Визуализацию распределения прогнозов (гистограмма, тепловая карта).
  5. Предоставляет возможность скачать результаты в Excel.

**Результат:** Рабочее приложение, доступное локально.

### 3. Этап: Тестирование, верификация и документирование

**Задача:** Провести финальное тестирование и подготовить отчёт.

- Проверить работу приложения на новых файлах, не участвовавших в обучении (например, данные студентов других групп или курсов).
- Провести качественный анализ:
  - Взять топ-10 студентов с наиболее высоким прогнозируемым риском.
  - Сравнить прогноз модели с реальными результатами следующей сессии (если данные доступны).
  - Оценить, насколько прогнозы модели совпадают с экспертным мнением кураторов.
- Оформить отчёт о проделанной работе, включающий:
  - Используемые алгоритмы и архитектуру.
  - Описание признаков и их важность.
  - Полученные метрики качества.
  - Инструкцию по установке и использованию приложения.
  - Рекомендации по улучшению системы (например, добавление данных о посещаемости, социально-экономических факторах).

**Результат:** Финальный отчёт и презентация проекта для заказчика.
